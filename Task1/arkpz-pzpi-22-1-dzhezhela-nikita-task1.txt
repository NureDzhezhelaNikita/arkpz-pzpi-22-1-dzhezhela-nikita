1. Бізнес вимоги
1.1. Передумови
Система управління автопарками повинна розв'язати проблеми, пов’язані зі зростаючими потребами бізнесу у зручному та ефективному управлінні великими автопарками. У сучасному світі компанії стикаються зі складнощами, такими як висока вартість обслуговування автопарку, недостатня прозорість даних про транспортні засоби та їх використання, а також труднощі у координації між різними підрозділами.
Що найменше, у місті, де я мешкаю, близько 8 компаній, загальний автопарк яких досягає 1000 одиниць транспортних засобів, за якими необхідний постій догляд цілих департаментів. Рішення які використовуються зараз – старі та неефективні, бо передбачають ручне звітування про кожен прецедент використання техніки, передачу ключів та постійний опис стану техніки, встановлення окремих GPS-трекерів, які складно інтегрувати з існуючими ERP та CRM системами.
1.2. Бізнес-можливості
Сьогодні ринок управління автопарками стрімко зростає через збільшення кількості транспортних засобів у бізнесі та необхідність підвищення операційної ефективності. Клієнти часто стикаються з такими проблемами:
* високі витрати на обслуговування через відсутність автоматизованого моніторингу стану транспортних засобів
* нестача прозорості у процесах управління та відсутність точних аналітичних даних
* обмеження у реальному часі для моніторингу автомобілів та їх систем
* відсутність інтеграції з існуючими бізнес-системами, такими як ERP або CRM
Конкурентний аналіз показує, що більшість існуючих рішень на ринку не мають достатньої гнучкості для адаптації до конкретних потреб замовника або застарілі. Наші конкуренті переваги:
* можливість адаптації для різних галузей
* збір даних у реальному часі
* масштабованість та інтеграція з іншими системами
* використання актуальних та захищених рішень
Ринок орієнтований на компанії логістичного сектору, служби доставки, орендні сервіси та інші організації, які активно використовують автопарки. Система відповідає сучасним тенденціям, таким як автоматизація, інтеграція з IoT та використання хмарних технологій. Це робить її привабливим рішенням для бізнесів, які прагнуть підвищити ефективність і зменшити витрати.
2. Концепція рішення
2.1. Окреслення концепції
Програмна система надає комплексне рішення для автоматизації процесів управління, моніторингу та аналізу автопарків середнього та великого бізнесу. Вона дозволяє отримувати дані про автомобілі у реальному часі, керувати їх системами, отримувати аналітичну інформацію через веб-інтерфейс та мобільні застосунки, а також забезпечує доступ до функцій через публічний API.
2.2. Головна функціональність
MF-1: Отримання даних про автомобіль у реальному часі
Отримання даних про автомобіль включає збір та обробку даних з IoT пристроїв автомобілів, таких як:
* локація транспортного засобу (за допомогою GPS)
* статус дверей (Аналітика внутрішніх систем автомобіля)
* рівень палива або заряд акумулятора (Аналітика внутрішніх систем автомобіля)
MF-2: Керування системами автомобіля у реальному часі
Дозволяє адмініструвати автопарк шляхом віддаленого керування системами автомобіля:
* блокування та розблокування дверей
* запуск або зупинка двигуна
* надсилання команд внутрішнім системам автомобіля
MF-3: Веб-система для отримання аналітичних даних
Надання інтерактивних звітів і аналітики, таких як:
* показники використання транспортних засобів
* статистика витрат на паливо
* прогноз обслуговування та ремонту
* дані про екологічність автопарку
MF-4: Веб-система для адміністрування автопарку
Функціонал включає:
* управління профілями водіїв
* планування технічного обслуговування
* організацію маршрутів і завдань
* перегляд історії дій та подій по кожному транспортному засобу
MF-5: Надання публічного API
Забезпечення сторонніх систем доступом до функцій системи:
* Доступ до інформації про транспортні засоби.
* Інтеграція з ERP або CRM системами.
* Використання API для автоматизації бізнес-процесів.
MF-6: Мобільний застосунок для внутрішнього використання співробітниками
Додаток надає можливість:
* переглядати стан доступних транспортних засобів у реальному часі
* отримувати доступ до автомобілів
* фіксувати події та сповіщення
* вести журнал технічного обслуговування
3. Рамки та обмеження проєкту 
3.1. Рамки первинного випуску 
Функціонал серверної частини:
SoIR-1: Реєстрація та авторизація користувачів системи з різними ролями.
SoIR-2: Зберігання даних про транспортні засоби.
SoIR-3: Адміністрування даних про транспортні засоби.
SoIR-4: Інтеграція з IoT-пристроями для збору даних про стан транспортні засоби.
SoIR-5: Публічне API для інтеграції з іншими системами.
SoIR-6: Мультимовна підтримка інтерфейсу, стандартизація форматів дати, часу та одиниць вимірювання.
Функціонал IoT-клієнта:
SoIR-7: Збір та обробка загальних даних про стан автомобіля: розташування автомобіля, стан двигуна, дверей, поточна швидкість.
SoIR-8: Збір та обробка даних про поточну сесію використання автомобіля: час використання, пройдений шлях, маршрут, середня швидкість, витрати пального.
SoIR-9: Синхронізація даних з сервером у різних режимах.
SoIR-10: Можливість конфігурування IoT клієнта: сервер для синхронізації, інтервали синхронізації, модуль обробки даних з CAN шини автомобіля.
SoIR-11: Наявність двох станів – активний та очікування.
SoIR-12: Обробка даних NMEA з GPS модуля.
3.2. Рамки наступних випусків 
Функціонал веб-складової:
SoSR-1: CRUD-операції для керування даними про транспортні засоби, маршрути та користувачів.
SoSR-2: Інструменти для аналізу використання автопарку.
SoSR-3: Перегляд інформації про транспортні засоби та маршрути у реальному часі на мапі.
Функціонал мобільної частини:
SoSR-4: Додаток для водіїв з функціоналом отримання доступу до автомобіля, звітів про стан автомобіля та завдань.
SoSR-5: Оповіщення про зміни маршрутів, технічний стан або інші події.
SoSR-6: Генерація QR-кодів для верифікації водіїв на базах та сервісах.
SoSR-7: Можливість перегляду історії маршрутів.
4. Бізнес-контекст 
4.1. Робоче середовище
Система буде працювати у хмарному середовищі, забезпечуючи доступ через веб-браузери (Chrome, Safari, Edge) та мобільні додатки (iOS, Android). Ключові аспекти робочого середовища:
* час відповіді сервера оптимізований для роботи в реальному часі
* безперебійний доступ до системи, незалежно від географічного розташування користувача
* інтеграція з ERP-системами компаній для аналізу операційних витрат та оптимізації бізнес-процесів
* у випадку тимчасової відсутності доступу до сервера, мобільний додаток фіксує дані для подальшої синхронізації після відновлення
Для реалізації використовуються:
Серверна частина: Severless рішення Supabase
Веб-складова: Next.js.
Мобільна частина: React Native.
IoT-пристрої: Платформи на базі ESP32 або Arduino.
СУБД: Реляційна база даних на базі PostgreSQL – SupabaseDB.
